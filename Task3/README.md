# Технический радар и роадмап трансформации "Будущее 2.0"

## 1. Технический радар

### Структура радара
- **ADOPT** - внедряем в продакшн
- **TRIAL** - тестируем на пилотах
- **ASSESS** - изучаем возможности
- **HOLD** - выводим из эксплуатации

### Технологические изменения

| Категория | HOLD (выводим) | ADOPT (внедряем) | TRIAL (пилотируем) | ASSESS (изучаем) |
|-----------|----------------|------------------|---------------------|-------------------|
| **Платформы** | SQL Server 2008<br>Apache Camel | PostgreSQL<br>Apache Kafka<br>Kubernetes | ClickHouse<br>Elasticsearch | Snowflake<br>DataBricks |
| **Инструменты** | Power Builder | GitLab CI/CD<br>Prometheus/Grafana<br>Terraform | Apache Airflow<br>Superset/Tableau | ArgoCD<br>Service Mesh |
| **Языки** | .NET Legacy | Java/Spring<br>Golang<br>Python<br>React | Node.js<br>TypeScript | Rust<br>Flutter |
| **Практики** | Монолит<br>Ручные деплои | Domain-Driven Design<br>Event-Driven<br>DevOps | Data Mesh<br>SRE | Chaos Engineering |

### Ключевые технологические решения

**Выводим:**
- **SQL Server 2008** → устарел, нет поддержки
- **Power Builder** → нет специалистов, плохой UX
- **Монолитная архитектура** → не масштабируется

**Внедряем:**
- **Kubernetes** → стандарт для микросервисов
- **Apache Kafka** → асинхронная интеграция доменов
- **PostgreSQL** → основная БД для доменов
- **React** → современный UI

## 3. Обоснование подхода

### Почему такая последовательность?

1. **Сначала фундамент** - без инфраструктуры невозможно развернуть новую архитектуру
2. **Затем quick win** - витрина данных решает острую проблему бизнеса
3. **Потом основные домены** - когда есть опыт и проверенная инфраструктура
4. **В конце отключение легаси** - когда всё работает на новой платформе

### Почему эти технологии?

**Kubernetes и микросервисы:**
- Независимое масштабирование доменов
- Изоляция сбоев
- Быстрые релизы

**Apache Kafka:**
- Асинхронная интеграция
- Надежная доставка событий
- Горизонтальное масштабирование

**PostgreSQL + ClickHouse:**
- PostgreSQL для операционных данных (OLTP)
- ClickHouse для аналитики (OLAP)
- Оптимальное решение для разных задач

**Domain-Driven Design:**
- Четкие границы ответственности
- Независимые команды
- Простота понимания системы
